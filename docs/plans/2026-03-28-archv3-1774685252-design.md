## 1. Objective and Success Criteria

Phase 1 adds three connected capabilities to the Kanban workflow:

1. Issue #65 (P0): introduce `AWAITING_OWNER` as a first-class state with distinct board behavior and a global TopBar badge.
2. Issue #66 (P1): show a compact actor handoff trail (`FlowStrip`) on each task card.
3. Issue #67 (P1): allow owner comments, with comment input gated to `AWAITING_OWNER` and thread visible in all states.

Success is not only visual completion. This phase is considered done when:

- state definitions and ordering are updated consistently in frontend types,
- backend status and task routes return the new data needed by UI,
- board sorting/urgency behavior for owner-waiting tasks matches specification,
- event posting remains backward compatible with current clients,
- new UI pieces are covered in tests (`tests/client/v2-phase1.test.tsx`),
- `npm run build` and existing tests pass.

## 3. Design Constraints and Non-Goals

### 3.1 Constraints

- `AWAITING_OWNER` must be placed between `EXECUTION` and `REVIEW_PENDING` in both pipeline/main flow state arrays.
- Design tokens must align with existing theme values; requested amber (`#F59E0B`) is the semantic accent for owner-waiting state.
- Existing API consumers must keep working while new UI paths are introduced.
- Task files remain the backend source of truth in `~/clawd/tasks`.
- The implementation must stay within the current architecture (Express-like API routes + React client).

### 3.2 Non-Goals

- No redesign of global navigation or unrelated tabs.
- No migration of storage model beyond appending events to existing `events.ndjson` files.
- No large refactor of polling architecture in this phase.

## 4. Alternative Approaches Considered

### Approach A: Incremental Compatible Extension (Recommended)

Extend current data contracts with additive fields, keep event endpoint compatibility, and add focused UI components without disrupting existing fetch flows.

Pros:

- lowest regression risk,
- straightforward rollout,
- preserves old clients while enabling new behavior.

Cons:

- temporary dual-format handling in event endpoint,
- minor fallback complexity in TopBar data sourcing.

### Approach B: Hard Contract Switch

Adopt only new event payload format and require immediate migration of all writers/readers.

Pros:

- simpler endpoint internals in the long term,
- cleaner single contract.

Cons:

- avoidable break risk,
- blocks phased rollout and increases review burden.

### Approach C: Client-Centric Derivation

Compute most new indicators on client from task lists/detail responses; minimal backend changes.

Pros:

- fewer server edits.

Cons:

- duplicated logic and inconsistent source of truth,
- increased client CPU and polling coupling,
- less reliable overdue badge behavior.

### Recommendation

Use Approach A. It is the only option that meets P0 stability expectations while still enabling all requested P1 features in one branch.

## 5. Detailed Design

## 5.1 Shared Types and Contracts (`src/lib/types.ts`)

### State Arrays

Add `AWAITING_OWNER` in both arrays at the exact requested position:

- `PIPELINE_STATES`: `... EXECUTION, AWAITING_OWNER, REVIEW_PENDING ...`
- `MAIN_FLOW_STATES`: same relative ordering.

This ordering is functional, not cosmetic. It drives column sequence and any state-progression logic that depends on array index.

### Global Status

Extend interface:

- `awaiting_owner_count: number`
- `awaiting_owner_overdue: boolean`

Both fields are additive and should be treated as optional-safe on old responses in UI code (guarded fallback path).

### Task Model Extensions

Ensure task typings expose fields needed by rendering/sorting:

- `deadline_at?: string | null`
- `updated_at?: string`
- `actors?: string[]`
- event shape includes `type`, timestamp, and payload with optional `actor/body`.

## 5.2 Backend Changes

## 5.2.1 `server/api/status.js`

Add owner-waiting aggregation while preserving current response payload shape.

Computation plan:

1. Enumerate known task directories under `~/clawd/tasks/`.
2. Parse each task’s state from status/source already used by status endpoint.
3. Filter tasks where `state === 'AWAITING_OWNER'`.
4. `awaiting_owner_count = filtered.length`.
5. `awaiting_owner_overdue = filtered.some(task => parsedDeadline(task) < now)` with safe parsing.

Safe parsing rules:

- if `deadline_at` absent, empty, or invalid: not overdue,
- no exception propagation; invalid values are ignored for overdue computation.

This allows badge pulse to be trusted without causing status endpoint instability from malformed files.

## 5.2.2 `server/api/tasks.js` — `GET /api/tasks/:id`

### Actor Derivation

Return `actors: string[]` as unique first-seen actor chain derived from events.

Algorithm:

1. Load events in chronological order.
2. For each event, extract `actor` from canonical location (payload or top-level, depending on stored format).
3. Normalize alias to canonical id.
4. Add to ordered set if first appearance.
5. Return ordered array.

Normalization map (minimum required):

- `main -> sokrat`
- `brainstorm -> brainstorm-claude`

Any unknown value should pass through unchanged, so future agents remain visible even before explicit mapping.

### Response Compatibility

`actors` is additive; existing consumers that ignore it remain unaffected.

## 5.2.3 `server/api/tasks.js` — `POST /api/tasks/:id/event`

Support both formats:

1. Legacy:

```json
{ "type": "...", "payload": { ... } }
```

2. New owner-comment path:

```json
{ "event_type": "USER_COMMENT", "actor": "owner", "body": "..." }
```

Canonicalization before append:

- if `event_type` exists, map to internal event object with `type = event_type` and payload/body fields.
- if legacy payload provided, preserve existing behavior.

Validation:

- reject if no effective event type,
- for `USER_COMMENT`, reject blank/whitespace-only `body`,
- return clear 4xx message on validation failure.

Write path:

- append canonical event JSON line to `~/clawd/tasks/<id>/events.ndjson`.

This preserves current storage semantics while enabling new UI writer behavior.

## 5.3 Frontend Changes

## 5.3.1 Kanban Board (`src/components/pipeline/KanbanBoard.tsx`)

### New Column Appearance

For `AWAITING_OWNER`:

- header icon: `⏳`
- header text: `Awaiting Owner`
- color token: amber `#F59E0B`
- column container style includes `bg-bg-void` and a 2px amber left stripe.

### Empty State Copy

When no cards are present in this column:

- show: `✦ Nothing waiting — you are clear`

Copy is scoped only to this column; others keep existing empty behavior.

### Sorting Behavior (Column-Specific)

Sort only for `AWAITING_OWNER` as:

1. valid `deadline_at` ascending,
2. fallback by `updated_at` descending when deadlines are absent/invalid,
3. stable deterministic order for equal keys (e.g., by id).

This guarantees predictable triage order and matches clarification `Q4`.

### Urgency Classification

For `AWAITING_OWNER` cards only:

- `overdue`: deadline valid and `< now`.
  - style: red left border + `animate-pulse-critical`
- `urgent`: deadline valid, `>= now`, and `< now + 1h`.
  - style: amber left border/glow + `animate-pulse-active`
- `normal`: no deadline or deadline >= 1h.
  - style: amber left border, no pulse.

Non-`AWAITING_OWNER` states keep current styling unchanged.

## 5.3.3 New `FlowStrip` Component (`src/components/pipeline/FlowStrip.tsx`)

### Props and Rendering Rules

Input:

- `actors: string[]` (already unique and ordered from backend, but component may still defensively dedupe).

Visual rules:

- 20x20 circular avatar per actor,
- background `bg-bg-overlay`, border `border-border-subtle`, centered emoji,
- arrow separator `→` with `text-text-disabled`,
- max 5 visible avatars,
- overflow indicator `+N` when actors length > 5,
- tooltip on hover: display agent name + role label.

### Emoji Mapping

Canonical map:

- `sokrat`/`main` = `🦉`
- `archimedes` = `⚙️`
- `platon` = `🏛️`
- `leo` = `🎨`
- `aristotle` = `📚`
- `herodotus` = `📜`
- `hephaestus` = `⚒️`
- `brainstorm-claude` = `🧠`
- `brainstorm-codex` = `💡`

Unknown actors use deterministic fallback (e.g., `•` or default emoji) plus tooltip text, never omitted.

## 5.3.4 TopBar Badge (`src/components/layout/TopBar.tsx`)

Add amber clickable badge immediately after agent count indicator:

- label format: `⏳ N`
- click target: Pipeline route (`/`)
- pulse when overdue flag true.

Data source order (from `Q2`):

### `CommentThread.tsx`

Responsibilities:

- filter/display `USER_COMMENT` events,
- render author identity, relative timestamp, and message body,
- maintain scrollable container with newest at bottom,
- show `No comments yet` when comment list is empty.

### `TaskDetail.tsx` Integration

At bottom of detail pane:

1. comment thread always,

Data loading should rely on the existing `GET /api/tasks/:id` route to fetch a coherent payload (task + events), avoiding dependence on separate non-existent routes.

## 6. Data Flow and Interaction Sequences

## 6.1 Board + TopBar Polling

1. Client polls `/api/status` and `/api/tasks` on existing cadence.
2. TopBar receives count/overdue from status, fallback only if absent.
3. Pipeline groups tasks by state including new `AWAITING_OWNER` column.
4. AWAITING_OWNER cards are sorted and styled via deadline rules.

## 6.2 Task Detail Open

1. User opens task detail.
2. Client fetches `GET /api/tasks/:id`.
3. UI renders events, derived actors (for card strip when list refreshes), and comments.
4. If state is `AWAITING_OWNER`, comment input is enabled.

## 6.3 Comment Submit

1. User enters text and submits.
2. Client validates non-empty body.
3. POST to `/api/tasks/:id/event` with new payload shape.
4. Backend canonicalizes and appends ndjson line.
5. Client refreshes detail/events and thread updates.
6. On error, toast appears and draft text remains.

## 7. Edge Cases and Failure Handling

### Deadline Ambiguity

- missing/invalid deadlines never crash sort,
- they are non-urgent by default,
- they still participate in fallback `updated_at desc` ordering.

### Actor Quality

- duplicate actors in source events collapse to first appearance only,
- alias normalization prevents emoji gaps for known historic names,
- unknown actors still display with fallback to avoid data loss.

### API Compatibility

- legacy event clients continue working,
- new owner-comment payload works immediately,
- mixed event history (old + new shapes) remains readable.

### Partial Deployments

If frontend with new badge ships before backend fields:

- fallback computation prevents blank UI,
- once fields appear, status becomes source of truth automatically.

### Write Failures

- event append failures return error and do not mutate UI state optimistically beyond draft preservation,
- user can retry without retyping.

## 10. Risks and Mitigations

1. Risk: schema drift in existing events file format.
   - Mitigation: tolerant parser + dual-payload acceptance + defensive actor extraction.
2. Risk: visual churn from adding pulses and amber accents.
   - Mitigation: scope animations only to awaiting-owner urgency and overdue badge.
3. Risk: inconsistent counts between status and client fallback.
   - Mitigation: fallback only when status fields are absent, not when present.
4. Risk: performance from extra derivation on every detail fetch.
   - Mitigation: actor extraction is linear in events and bounded by task history size; acceptable for Phase 1.
