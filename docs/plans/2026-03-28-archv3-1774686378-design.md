## 3. Approaches Considered

### Approach A: Additive, Backward-Compatible Extension (Recommended)

Add new fields/components without breaking existing API contracts.

Pros:

- Lowest regression risk.
- Fits existing architecture and polling model.
- Enables phased rollout and easier verification.

Cons:

- Some temporary dual-format handling.

### Approach B: Strict New Contract Cutover

Only support new event payload and immediately migrate all callers.

Pros:

- Cleaner long-term API surface.

Cons:

- Higher break risk for existing callers/components.
- Larger single-step change footprint.

### Approach C: Client-Derived Everything

Keep backend minimal and derive new indicators entirely from existing task responses.

Pros:

- Fewer backend edits.

Cons:

- Duplicated business logic in UI.
- Inconsistent source of truth for overdue/count data.

### Recommendation

Use Approach A.

## 4. Detailed Design

## 4.1 Types and Shared Contracts (`src/lib/types.ts`)

### Pipeline state updates

- Add `AWAITING_OWNER` between `EXECUTION` and `REVIEW_PENDING` in:
  - `PIPELINE_STATES`
  - `MAIN_FLOW_STATES`

### Global status updates

Extend `GlobalStatus`:

- `awaiting_owner_count: number`
- `awaiting_owner_overdue: boolean`

### Task fields needed for sorting/flowstrip/comments

Ensure task/event typing supports:

- task-level `deadline_at?: string | null`
- task-level `updated_at?: string`
- task-level `actors?: string[]`
- event-level `event_type` + `actor` + timestamp + optional `body`

## 4.2 Backend Design

## 4.2.1 Status aggregation (`server/api/status.js`)

Add computation from task files in `~/clawd/tasks/`:

1. Load task statuses as currently done for other counters.
2. Filter `state === 'AWAITING_OWNER'`.
3. Set `awaiting_owner_count` to filtered length.
4. Determine `awaiting_owner_overdue` by checking any valid deadline `< now`.

Deadline parsing rules:

- Missing/invalid deadline is not overdue.
- Parsing failures do not fail the endpoint.

## 4.2.2 Tasks detail/event endpoint (`server/api/tasks.js`)

### Actors extraction for FlowStrip

On task detail response, provide `actors[]` derived from `events.ndjson`:

- chronological order,
- first-seen unique actors,
- ignore malformed rows,
- keep unknown actor ids as-is.

### POST `/api/tasks/:id/event`

Support both request shapes:

- Existing: `{ type, payload }`
- New comment path: `{ event_type: 'USER_COMMENT', actor: 'owner', body: '...' }`

Canonical write behavior:

- Normalize both shapes into one event object.
- Append one JSON line to task `events.ndjson`.
- Validate `USER_COMMENT` body is non-empty.

## 5. Data Flow

1. Polling fetches tasks/status.
2. Status returns `awaiting_owner_count` and `awaiting_owner_overdue`.
3. TopBar renders owner badge from status.
4. Task detail fetch includes events and derived `actors[]`.
5. TaskCard renders `FlowStrip` from `actors[]` and urgency from deadline fields.
6. Owner submits comment -> API appends `USER_COMMENT` event -> subsequent poll refreshes thread/card indicators.

## 6. Error Handling and Edge Cases

- Invalid/missing deadlines: treated as non-urgent, not overdue.
- Unknown actor ids: still shown with fallback emoji and tooltip text.
- Empty comment body: client validation + server 4xx validation.
- Event file write failure: return 5xx with stable JSON error shape.
- Backward compatibility: maintain legacy `addTaskEvent(type,payload)` call path.
